# Архитектура RefurbChain

## Компоненты
- Smart contract `DevicePassportRegistry`: хранит список доверенных сервисов и on-chain доказательства отчётов.
- Backend `Express`: принимает форму диагностики, считает хэши, пишет отчёт в локальную БД и отправляет транзакцию в контракт.
- Frontend `React/Vite`: интерфейс сервисного центра и страница верификации для покупателя.
- Local JSON DB: off-chain хранение полной версии отчёта.

## Поток данных
1. Сервисный центр вводит данные устройства.
2. Frontend отправляет JSON на backend.
3. Backend формирует `deviceHash = keccak256(serial:salt)`.
4. Backend формирует `reportHash = keccak256(JSON(report))`.
5. Backend вызывает `registerReport()` в смарт-контракте.
6. Контракт эмитит `ReportRegistered`.
7. Backend сохраняет off-chain запись и генерирует QR для `/verify/{reportId}`.
8. Покупатель открывает страницу проверки и получает данные из backend + блокчейна.

## Что on-chain
- deviceHash
- reportHash
- serviceAddress
- timestamp
- status
- flags
- metadataURI

## Что off-chain
- модель
- результаты диагностики
- маскированный серийный номер
- человекочитаемые детали отчёта
- QR и short-link mapping

## Модель угроз
- Подделка PDF/JSON отчёта вне системы.
- Попытка записи отчёта неавторизованным сервисом.
- Компрометация сервисного ключа.
- Подмена QR для другого устройства.
- Раскрытие серийного номера в публичной сети.

## Меры защиты
- Role-based access control через AccessControl.
- Обезличивание устройства через хэш serial+salt.
- Хранение в блокчейне только доказательства, а не всего документа.
- Возможность revoke сервисного адреса администратором.
